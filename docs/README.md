# DOCS

本文档说明仓库内各类文档的定位，避免内容重叠与边界混乱。

## 文档分层

- **README（入口）**：告诉人和 AI 项目是什么、怎么跑、先看什么。
- **ARCHITECTURE（总图）**：只描述系统大结构、模块边界、依赖规则与不变量。
- **FEATURES（功能）**：每个新功能一份文档，描述目标、用例、约束、验收标准。
- **ADR（决策）**：记录跨功能、长期有效的架构与工程决策。
- **TESTING / RUNBOOK / STYLEGUIDE（执行规范）**：测试、运维、代码风格规则。

## ADR 与 FEATURES 的关系

- `docs/features/*.md` 回答：**这次具体要做什么**。
- `docs/adr/*.md` 回答：**为什么长期要这么做**。

两者关系是：

1. 一个功能文档可以不产生 ADR。
2. 一个 ADR 可能影响多个功能文档。
3. 当功能改动触发“长期规则变化”时，必须补 ADR。

## 什么时候只写 Feature

满足以下条件通常只需写 `docs/features/<feature>.md`：

- 改动范围局部，且不改变模块边界。
- 不改变公共 API 契约与依赖方向。
- 不改变长期技术路线，只是在既有路线下交付功能。

## 什么时候必须写 ADR

出现以下任一情况，必须新增或更新 ADR：

- 变更架构风格或模块边界。
- 变更跨模块依赖方向。
- 变更公共 API 设计原则。
- 引入长期约束（例如统一数据源策略、持久化策略、交易规则基线）。
- 一个决策会影响多个后续功能。

## 推荐工作流

1. 先看 `ARCHITECTURE`，确定模块落点。
2. 创建 `docs/features/<feature>.md`，明确目标与验收。
3. 实现代码并补测试。
4. 若发现是长期决策，补 `docs/adr/NNN-*.md`。
5. 更新 `DECISIONS.md` 与相关执行规范文档。
