# FEATURE: ops_requirements

## Goal

- 把 SysQ 日常运营闭环的硬要求写进项目文档，减少口头规则丢失。
- 让后续人类或 AI 执行 daily / weekly 流程时，默认知道哪些检查与输出是不可省略的。

## Use Cases

- 用例 1：AI 触发每日流程前，先检查数据日期、缺口、字段和空值质量，避免用脏数据继续生成计划。
- 用例 2：训练完成后，操作者必须能看到训练指标，而不是只看到模型文件落盘。
- 用例 3：回测完成后，操作者必须能看到回测指标和可追踪日志，便于排障与复盘。
- 用例 4：盘前候选输出必须可操作，不能只有股票代码，必须给出分数/排序依据、权重、股数和参考价格。

## API Change

- 是否新增 API：否。
- 是否修改现有 API：否。
- 变更点列表：
  - 文档层约束补充到 `AGENTS.md`、`docs/ARCHITECTURE.md`、`docs/TESTING.md`、`docs/RUNBOOK.md`、`ROADMAP.md`。

## UI

- 无。

## Constraints

- 这些要求默认适用于每日流程、训练流程、回测流程。
- 若暂时无法满足，应在输出中明确标记 blocker，而不是忽略。

## Done Criteria

- 文档中明确记录以下硬要求：
  - 数据健康检查是前置步骤
  - 训练必须输出指标摘要
  - 回测必须输出关键指标与交易日志
  - 候选输出必须包含分数/权重/股数/参考价格
- 文档之间无明显冲突。

## Test Plan

- 文档变更，无代码测试。
- 由后续实现任务在流程中逐步对齐这些要求。

## Rollback Plan

- 按 commit 回滚文档修改。

## Notes

- 这不是一次性实现清单，而是 SysQ 运营闭环的长期约束。
