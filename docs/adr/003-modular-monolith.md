# ADR 003: 采用模块化单体架构 (Modular Monolith)

**状态**：已接受 (Accepted)
**日期**：2026-03-14

## 背景 (Context)

SysQ 需要支持从数据处理、模型训练到实盘交易的完整链路。作为一个量化系统，它具有以下特点：
1.  **链路长**：涉及离线数据、实时数据、模型、回测、实盘等多个环节。
2.  **数据耦合**：不同环节共享同一套数据 Schema 和特征定义。
3.  **团队规模**：初期由少量核心开发人员维护，不需要复杂的微服务治理。

我们需要决定系统的顶层架构风格。

## 决策 (Decision)

我们决定采用 **模块化单体 (Modular Monolith)** 架构。

具体定义为：
- **物理上**：单一 Git 仓库，单一 Python 包 (`qsys`)。
- **逻辑上**：强制划分领域模块（Data, Model, Strategy, Trading），模块间通过明确的 Python API 交互，禁止私有实现跨模块调用。
- **部署上**：同一套代码可部署为训练节点（只跑 `run_train.py`）或交易节点（只跑 `run_daily_trading.py`）。

## 替代方案 (Alternatives)

### 1. 微服务架构 (Microservices)
- **方案**：将数据服务、模型服务、交易服务拆分为独立进程/容器，通过 HTTP/RPC 通信。
- **缺点**：运维复杂度过高（需要服务发现、熔断、分布式追踪），且 Python 进程间传输大量 Dataframe 数据会有显著的序列化开销。
- **结论**：在当前阶段属于过度设计。

### 2. 脚本大杂烩 (Script Collection)
- **方案**：没有核心包，只有一堆 `scripts/do_x.py`，逻辑散落在各个脚本中。
- **缺点**：逻辑无法复用（回测和实盘是两套逻辑），维护成本极高，容易产生“行为偏离”。
- **结论**：不可接受，必须有核心业务层 `qsys`。

## 结果 (Consequences)

### 正面影响
- **开发效率**：代码跳转、重构、IDE 提示非常流畅。
- **部署简单**：`pip install .` 即可分发。
- **一致性**：回测和实盘直接复用同一份 `qsys.strategy` 代码，彻底消除“回测实盘不一致”的常见痛点。

### 负面影响
- **边界腐化风险**：如果没有严格的代码审查，容易出现模块间循环依赖。
- **依赖膨胀**：所有依赖都在一个 `requirements.txt` 里，可能导致镜像体积较大。

### 缓解措施
- 在 CI 流程中加入依赖检查，禁止反向依赖（如 `data` 依赖 `strategy`）。
- 编写 `ARCHITECTURE.md` 明确模块边界。
